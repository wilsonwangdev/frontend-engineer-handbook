## Context

[SPEC-0001](../0001-scope-and-audience/spec.md) 把"AI 可委托清单"列为
手册的核心立场之一。第 0 章给了速查版，第 1 章会给完整版。但**究竟以
什么标准把一项任务归入"可委托"**，需要一份可执行的判断依据——否则未来
扩展清单时会陷入主观争论，也会让读者怀疑这些归类是"AI 自吹"。

本 SPEC 就是这份判断依据。任何关于"X 任务该不该委托"的争论，先回到这里
对照标准。

## Decision

### 判断维度

一项任务被归入"可委托 AI"档，必须**同时**满足以下 4 项（D1–D4 为硬性）：

#### D1：验收标准可被人类快速验证

> 你能在 5 分钟内判断 AI 的输出是对是错。

- 单元测试可断言（"这个函数对这 10 个输入返回什么"）
- 视觉可比对（"这个组件渲染后长这样"）
- 行为可复现（"点这个按钮应该 navigate 到 /xxx"）

反例：架构决策的"对错"要 6 个月后才显现 → 不可委托。

#### D2：上下文需求在 prompt 容量内

> 任务所需的全部上下文能塞进一个合理长度的 prompt（或通过 AGENTS.md +
> 项目结构隐式传达）。

- 写一个独立组件：上下文 = 设计稿 + props 类型 → 可容纳
- 写一个跨 30 个文件的重构：上下文 = 整个仓库 → 超出可靠范围（即使
  context window 容纳了，AI 也难维持长程一致性）

#### D3：错误代价低且可回滚

> AI 出错时不会造成长期或难以发现的后果。

- 样板代码出错：编译/测试当场报错，本地修复 → 低代价
- 数据库迁移 SQL 出错：可能损坏生产数据 → 高代价
- 安全相关代码出错：可能数月不被发现 → 高代价

D3 不是绝对的——好的工具链（类型检查、测试、code review）能降低代价，
让更多任务进入可委托范围。

#### D4：所需"品味"在文档化的规范内

> AI 不需要做主观审美/风格判断；规范在 AGENTS.md / specs / 设计系统中
> 已经定义清楚。

- 命名风格：项目有 ESLint/oxlint 规则 → 可委托
- 选择 Zustand 还是 Redux：架构决策，需要权衡 → 不可委托
- 实现 TierBadge 组件：设计系统已定义 → 可委托

#### D5（建议性）：警惕"委托后会让你变笨"的任务

> 不是硬性排除项，而是给读者的提醒。

某些任务通过了 D1–D4，技术上完全可以委托给 AI，但**长期反复委托会让人
失去关键判断力**。例如：

- 调试 Promise 链 / 异步 bug：调试经验是工程师的核心能力之一
- 阅读不熟悉的代码：理解力训练
- 写第一个 React 组件：心智模型的建立
- 排查性能问题：profiling 直觉

这些任务**满足 D1–D4 也算"可委托"**，但本手册会在对应章节附上一条警示：

> ⚠ 这项可以委托，但前 N 次建议自己做——本手册希望你保留这部分能力。

D5 是手册作者的立场，不是强制规定。读者完全可以选择全部委托。
这条维度的存在是为了对读者负责，而不是替读者决定。

判断 D5 是否触发的简单问题：

> 如果你过去 3 年只让 AI 做这件事，现在让你 30 分钟内独立完成，你做得到吗？
> 做不到 → 触发 D5 警示。

### 委托清单分类法

通过 5 项判断的任务，按**风险半径**进一步分级：

| 等级       | 含义                      | 示例                         |
| ---------- | ------------------------- | ---------------------------- |
| **Tier 1** | 完全委托，不必逐行 review | commit message、变更日志草案 |
| **Tier 2** | 委托后 diff review        | 样板组件、单元测试骨架       |
| **Tier 3** | 委托后逐行 review + 测试  | 业务逻辑、SQL 查询           |
| **Tier 4** | 用 AI 启发但人写最终版    | 性能优化思路、错误信息解读   |

"可委托 AI"在手册中的默认含义是 **Tier 1–3**。Tier 4 应表述为"AI 辅助"
而非"委托"。

### 维护机制

#### 何时更新清单

任何对清单的增删都必须：

1. 在本 SPEC 末尾追加一条变更记录（带日期 + 5 项判断的具体回答）
2. 同步更新 `content/chapter-00/index.mdx` 的速查版
3. 同步更新 `content/chapter-01/` 的完整版
4. 不允许"个人感觉应该加上"——必须有具体的失败/成功证据

#### 何时降级或移除

如果一项过去"可委托"的任务因为出现了新的失败模式（如发现 AI 在生成的代码
中频繁引入特定漏洞），降级或移出清单。移出时在变更记录里注明原因。

#### 季度复核

每季度抽样：

- 选 3 项 Tier 1–2 的任务，让 AI 实际执行，验证当前主流模型仍能稳定通过
- 选 3 项"暂时不要委托"的任务，验证它们是否仍然不可委托（也许进步了）
- 把结论写入 journal/

### 与"必学 / 仍需理解 / 可委托"三档的关系

不是同一回事，但相关：

- **三档**（[SPEC-0003](../0003-table-of-contents/spec.md)）描述**知识点**
  的必要程度
- **委托清单**（本 SPEC）描述**任务**的可委托程度

例：React 渲染规则是"必学"知识点 / 让 AI 写一个 React 组件是"可委托"任务。
**理解必学，敲字可委托。**

## Consequences

- 清单的每一次扩展都需要走 4 项硬性 + 1 项建议判断，写作变慢但可信度提高。
- 第 1 章必须把完整清单按这 4+1 项展开，否则失去判断依据的意义。
- 季度复核增加运营负担——但这是手册"时效性"承诺的兑现方式。
- D5 作为建议性维度而非硬性排除项，避免了"凭什么手册替我决定"的争议，
  同时保留了对读者长期能力的责任感。

## Alternatives considered

- **只用 D1（验收可验证）一条**：太宽松，会把"安全代码"也归入可委托。
  放弃。
- **完全主观，不写判断依据**：失去手册立场的可信度。放弃。
- **抄 GitHub Copilot 的"安全使用清单"**：那是工具厂商的免责声明，不是
  独立的工程判断。放弃。
