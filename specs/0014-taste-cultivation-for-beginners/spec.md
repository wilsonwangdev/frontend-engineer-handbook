## Context and Problem Statement

[SPEC-0001](../0001-scope-and-audience/spec.md) §AI 时代取舍 把"品位"
（taste）列为工程师价值的新落点之一，与判断、上下文管理、验证并列。
但"品位"对初学者是最难建立的门槛——它通常被认为是"看过足够多代码
后自然形成的直觉"，初学者既缺这个量、也缺辨别"哪些代码值得看"的
能力，于是品位看起来像玄学，被默认为天赋而非可训练能力。

在 AI 时代这个问题更急迫：

- AI 能高速产出**可工作但平庸**的代码，初学者很难分辨平庸与优雅
- 初学者容易**对复杂迷恋**——把抽象层数 / 设计模式数量 / 代码精巧度
  误当成"高水平"
- 初学者容易**被反模式误导**——网上大量"教程"本身就是反模式集合，
  AI 训练数据也包含大量历史反模式
- 一旦在前几年形成错误品位，后续要花数倍时间扭转——品位是有路径
  依赖的

2026-05-21 会话用户明确表态："如何帮助新手更早地培养品位意识是本
手册的核心目的之一，要尽早避免对复杂的迷恋或被反模式误导，培养对
优雅设计和架构的品位"。

需要把"品位可训练"立成手册的写作基调，并提供**具体可执行的训练触点**，
否则品位永远停在第 1 章的口号里，每章作者各凭直觉决定要不要带初学者
分辨好坏，结果就是初学者读完手册仍然不知道好品位长什么样。

## Decision Drivers

- **品位可训练**是手册的硬立场——拒绝"天赋论"
- **路径依赖**：初学者前几年形成的判断习惯决定上限，越早干预越值
- **AI 时代加剧**：可工作 ≠ 优雅；初学者需要"工作之外的辨别能力"
- **嵌入而非堆砌**：品位训练触点跟随具体场景出现，不为讲而讲
- **与 [SPEC-0011](../0011-content-accessibility-standard/spec.md) 一致**：
  抽象前必有具体例子；"品位"本身就是抽象概念，开篇必须先有锚点
- **跨会话防遗失**：基调写入 specs/，新会话扫入口表自动捕获
- **诚实**：作者不假装自己品位完美——这本身就是一种品位示范

## Considered Options

- **A. 立 SPEC 设品位训练四触点（采纳）**：把"对照 / 反思 / 命名 /
  透明化"作为可执行触发器，每章遇到设计判断时主动应用
- **B. 独立成章（如新增"品位训练手册"章）**：违反 [SPEC-0001](../0001-scope-and-audience/spec.md)
  实践定位；脱离场景讲品位最容易退化成口号
- **C. 只放第 1 章一段宣言**：初学者读完即忘；无法跨章持续训练
- **D. 完全不立 SPEC，靠 [SPEC-0011](../0011-content-accessibility-standard/spec.md) /
  [SPEC-0013](../0013-design-principles-as-undercurrent/spec.md) 自然带过**：
  那两个 SPEC 管"读得懂 / 设计判断该不该点破"，不管"如何辨别好坏"——
  品位训练是独立维度

## Decision Outcome

选 **A**。建立"品位训练暗线"由四块组成：

### 1. 立场宣言：品位是可训练的

第 1 章在已有"AI 可委托清单"主线之外，加一段**简短的品位立场宣言**
（≤ 10 行），明确陈述：

- 品位不是天赋，是经过大量"对照 + 反思"训练形成的直觉
- AI 时代品位反而更值钱——AI 能写可工作代码，**辨别可工作 vs 优雅**
  是人的判断
- 本手册的承诺：每章在合适位置带读者做一次"对照"，不留口号

宣言放第 1 章是因为它是世界观陈述（与"AI 可委托清单"同性质）。
宣言本身不展开如何训练——那是后续四触点的事。**宣言 ≤ 10 行**，
避免章变重。

### 2. 品位训练四触点（每章遇到具体设计判断时主动应用）

写到一个有"好 / 坏" 之分的设计场景时，问自己能否触发以下任一项：

#### 触点 A：对照（contrast）

把同一个问题的**两种以上写法并列展示**，让读者亲眼看到差异。

要点：

- 对照对象必须可工作——不是"对 vs 错"，是"两个都能跑、但一个更好"
- 每个对照配一句"为什么这个更好"——避免读者只记表面差异
- 不超过 3 个版本——多了读者抓不住主线

例：讲 React 状态管理时，并列「prop drilling 5 层 / context / 状态
提升 + 局部 store」三种实现同一个 todo list，每个版本配 1-2 句"在
什么场景下它最合理 / 它的代价是什么"。

#### 触点 B：反思（reflection）

讲一个技术决策时，**显式说"我以前这么做、后来发现不好、现在这么做
因为 X"**。把作者自己的品位演化暴露给读者。

要点：

- 用第一人称、具体场景——不要写"很多人会这么做"这种抽象指代
- 重点不是炫"我现在很厉害"，是展示**判断标准如何形成**
- 失败 / 翻车 / 后悔类反思优先于成功类——更稀缺、更可信

例：「我以前喜欢把所有组件 props 都做成 optional 加默认值——直到
某次重构发现这让"必须传 vs 可不传"完全模糊，调用方根本不知道哪些
是关键参数。现在我倾向必填字段就明确必填，让类型系统替我把关。」

#### 触点 C：命名反模式（name the anti-pattern）

给常见的"看起来高级但其实糟糕"的写法**起一个具体名字**，让读者
下次见到时能识别。

要点：

- 名字要具体到能搜索——避免"过度设计"这种抽象词
- 配 1-2 行"它为什么诱人 / 它的代价"——光命名不解释没用
- 优先用业界已有名词——AbstractFactoryFactory、God Object、
  shotgun surgery、premature abstraction、wrapper hell
- 没有现成名字时再自创——避免造词通胀

例：「**抽象税（abstraction tax）**——为了"以后可能要扩展"提前
加的抽象层，往往在真要扩展时反而成为障碍。识别信号：抽象只有一个
实现 + 没有真实的第二个使用场景。」

#### 触点 D：透明化作者判断（transparency）

把"为什么这么选"明说出来，**包括没选某个流行方案的理由**。

要点：

- 不要假装"这个方案显然最好"——读者不傻，看得出有替代
- 显式说"我没选 X 因为 Y"——比假装 X 不存在更培养品位
- 偏好不等于真理——明说"这是偏好，不是定论"
- 避免单纯诉诸权威（"业界都这么做"）——讲清楚业界为什么这么做

例：「本手册推荐 Zustand 而非 Redux Toolkit，不是因为 Zustand 更
"先进"——RTK 在大型团队、需要严格规范的场景仍然合理。我推 Zustand
是因为本手册的预设场景是中小型项目 + AI 协作，那里 Zustand 的
"少即是多" 心智更省心。」

### 3. 反模式速查（持续追加）

收拢全书命名过的反模式（触点 C 产物）成一个整站索引，作为初学者
的"品位免疫清单"。位置选第 10 章末尾或附录 D 实战避坑录——与
[SPEC-0005](../0005-companion-tracks-and-test-strategy/spec.md)
附录 D 协同：

- 附录 D 收**作者亲自踩过的具体坑**（实例驱动）
- 本 SPEC 反模式速查收**有名字的常见反模式**（命名驱动）
- 两者交叉时优先附录 D，反模式速查只放"还没真踩到但已识别的常见
  类别"

种子清单（写作过程中持续追加）：

| 反模式名                         | 一句话识别信号                               | 出现章节     |
| -------------------------------- | -------------------------------------------- | ------------ |
| 抽象税（premature abstraction）  | 抽象只有一个实现且没有真实第二场景           | 第 5 / 6 章  |
| Wrapper hell                     | 一个简单值被层层包装，调用栈每层都只转发     | 第 5 章      |
| God object / God component       | 一个文件 / 组件承担了三件以上不相关职责      | 第 5 章      |
| Shotgun surgery                  | 改一个需求要散弹式改 N 个文件                | 第 6 / 10 章 |
| 复杂度炫耀（complexity flexing） | 用了花哨语法 / 模式但本质问题没简化          | 全书         |
| AI 平庸代码（AI-mediocre code）  | 能跑、命名平庸、过度防御、与代码库风格不匹配 | 第 9 章      |

### 4. 与第 1 章「AI 可委托清单」的边界

[SPEC-0013](../0013-design-principles-as-undercurrent/spec.md) 已经
讨论过"是否在第 1 章加反向清单"——结论是先靠每章嵌入累积，未来再
评估。本 SPEC 的立场宣言（§1）是**唯一**允许在第 1 章主动加的内容，
其他训练动作都嵌在后续章节。这条规则避免第 1 章被同时压上"AI 可
委托" + "设计判断反向清单" + "品位立场" 三件事。

## Consequences

- **好**：品位从口号变成可训练的四触点；初学者每章都能做一次对照 /
  反思 / 命名 / 透明化的训练
- **好**：跨会话不丢；新会话扫 specs/ 入口表自动捕获
- **好**：与 [SPEC-0011](../0011-content-accessibility-standard/spec.md) /
  [SPEC-0013](../0013-design-principles-as-undercurrent/spec.md) 形成
  内容四层标尺：定位（[SPEC-0001](../0001-scope-and-audience/spec.md)）
  / 写法（SPEC-0011）/ 深度（SPEC-0013）/ 品位（本 SPEC）
- **好**：诚实暴露作者判断（触点 B / D）本身就是品位示范
- **不好**：触点 B 反思要求作者持续暴露翻车经历——心理门槛 + 内容
  开放性都会增加
- **不好**：每章作者多一道"是否触发四触点之一"的反射检查
- **不好**：反模式速查表前期会很短——与 [SPEC-0013](../0013-design-principles-as-undercurrent/spec.md)
  第 10 章索引表同性质，不要为填满而填
- **待还的债**：写第 2 章时按本 SPEC 验证一遍，看四触点是否真嵌得
  进去 + 不让章节臃肿

## Pros and Cons of the Options

| 选项                                  | 好处                                               | 代价                                                 |
| ------------------------------------- | -------------------------------------------------- | ---------------------------------------------------- |
| **A. 四触点 + 反模式速查（采纳）**    | 跨会话不丢；嵌入贴近实战；与现有 SPEC 形成完整标尺 | 作者多一道反射检查；触点 B 心理成本                  |
| B. 独立成章                           | 系统性强                                           | 违反 SPEC-0001 实践定位；脱离场景讲品位易退化为口号  |
| C. 只放第 1 章宣言                    | 写起来轻                                           | 读完即忘；无法跨章持续训练                           |
| D. 不立 SPEC，靠 0011 / 0013 自然带过 | 零结构开销                                         | 品位训练是独立维度，那两个 SPEC 不直接覆盖；浓淡不一 |

## Confirmation

如何确认本 SPEC 被遵守：

1. **每章 PR 前作者自查**：commit body 或 PR 描述简短回答——
   "本章触发了哪几类品位训练（A 对照 / B 反思 / C 命名反模式 /
   D 透明化）？放在哪几节？"。零触点也算合格答案（不强求每章都
   有），但要显式说明"本章为何不触发"
2. **反模式速查表持续追加**：每次某章用触点 C 命名一个反模式时，
   同步追加一行
3. **季度复核**（与 [SPEC-0006](../0006-agent-skills-system/spec.md) /
   [SPEC-0010](../0010-progressive-ssg-distillation/spec.md) /
   [SPEC-0011](../0011-content-accessibility-standard/spec.md) /
   [SPEC-0013](../0013-design-principles-as-undercurrent/spec.md) 同周期，
   2026-08 首次）：
   - 检查四触点是否真在各章自然出现，还是被作者回避
   - 评估反模式速查表是否需要重组（按章节 / 按严重度 / 按场景）
   - 评估第 1 章宣言是否需要更新（基于初步章节的实操经验）
4. **不引入自动化校验**——与 [SPEC-0008](../0008-pre-action-reflexive-checklist/spec.md) /
   [SPEC-0011](../0011-content-accessibility-standard/spec.md) /
   [SPEC-0013](../0013-design-principles-as-undercurrent/spec.md) 一致

### 边界澄清

本 SPEC **不是**：

- 一份"什么是好代码"的权威定义——品位本身就是有偏好的，本 SPEC 只
  保证作者把偏好显式化
- 一份反模式百科——业界已有 refactoring.guru / wiki，无需重复
- 让作者炫耀自己品位的舞台——触点 B 反思优先翻车经历，不是成功秀

本 SPEC **是**：

- 一条"品位可训练"的硬立场陈述
- 四个嵌入式训练触点的反射触发器
- 一张帮初学者免疫常见反模式的速查清单
