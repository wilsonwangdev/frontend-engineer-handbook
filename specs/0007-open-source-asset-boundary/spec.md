## Context

写作中曾把"在自媒体（Twitter / 小红书 / 知乎等）公告"这种带个人推广性
质的句子写进 `docs/DEPLOY.md`（已修复）。这暴露了一个元规则缺失：**没有
明文标准来判断"什么内容属于开源资产、什么内容属于作者私事"**。

开源仓库的内容会被任何 fork 者继承。一旦个人推广策略、私人时间安排、
未授权信息进入仓库，不仅污染公共资产、降低项目可移交性，**长期还会
让 agent 在没有人监督时反复越界**——因为这种内容没有客观标准，agent
只能凭"看起来像项目内容"来判断，必然误伤。

本 SPEC 锁定边界，作为对未来所有写入开源仓库内容的元约束。

## Decision

### 测试法（写每段内容前必过）

新写入的任何内容（commit / 文件 / 注释），过两道测试：

**Test A — fork 测试**：

> 如果项目作者退出维护、被任何人 fork 走，这段内容是 fork 者**应当
> 继承的事实**、还是**作者个人立场**？

如果是后者 → 不进项目。

**Test B — 客观性测试**：

> 这段内容是否引用了某个**具体个人**的偏好、计划、社交账号、商业目标
> 或推广策略？

如果是 → 不进项目。

### 黑名单（一律不写入项目）

| 类别                        | 例子                                                                   |
| --------------------------- | ---------------------------------------------------------------------- |
| 推广渠道与社交账号          | "在 Twitter / 小红书 / 知乎 公告"、个人 Twitter handle、播客链接       |
| 个人时间承诺                | "周末上线"、"6 月底前 v1"、"每周更新"                                  |
| 个人 OKR / 商业目标         | "目标 1k star"、"年内打通付费层"                                       |
| 未授权的他人信息            | 团队成员真名 / 邮箱、未公开的 contributor 列表                         |
| 私有联系方式                | 个人邮箱（除 LICENSE 必要字段）、电话、即时通讯账号                    |
| 带主观情绪的呼吁            | "快来 star"、"求转发"、"加油干"                                        |
| 作者个人故事 / 履历         | "我在 X 公司学到 Y"、"以前在 Z 项目踩过这个坑"（除非匿名化为通用陈述） |
| 未授权第三方推荐 / 利益声明 | "推荐用 X 服务（联盟链接）"                                            |

### 灰名单（需特殊判断才能进）

| 类别                          | 进入条件                                                                            |
| ----------------------------- | ----------------------------------------------------------------------------------- |
| 同作者的其他开源项目链接      | 仅在 README "相关项目" 出现，明示关系，不进正文                                     |
| 作者个人技术博客              | 按 [SPEC-0002](../0002-content-source-admission/spec.md) 准入规则判断；多数情况不收 |
| 作者署名的客观经验提炼        | 必须脱敏 / 通用化（"我"改为"工程师"），并落 SPEC 或 journal                         |
| 作者 GitHub 用户名 / 仓库路径 | 作为归属与反馈渠道允许出现；不作为推广点                                            |

### 白名单（理所当然属于开源资产）

不需要审核，可直接写入：

- 项目自身的工程决策、技术选型、SPEC、journal、ROADMAP
- 客观的项目状态（已完成 / 进行中 / 规划中）
- 中性的贡献入口（GitHub Issues、Discussions）
- License / Copyright（版权归属是法律事实，不是推广）
- 协作约定（commit conventions、CI 规则、PR 模板）
- 工程化参考资料引用（如本项目对 agent-master-handbook 的引用）
- 公开的、客观的工具链 / 框架选型理由

### 个人内容的正确去处

被 Test A 或 B 排除的内容**不是不能存在**——只是不该在开源仓库里：

| 内容类型                     | 应去的地方                                     |
| ---------------------------- | ---------------------------------------------- |
| 作者推广 / 发布策略          | 用户级笔记 / Notion / 私有 repo                |
| 作者时间表 / 个人 OKR        | 用户级日历 / 项目管理工具                      |
| 作者私人 memory              | Claude `/memory` 用户级文件、Cursor 用户 rules |
| 作者在某项目的踩坑（个人化） | 个人技术博客 / 笔记                            |

写完一段内容如果发现它"违反 Test A/B 但又有价值"——**不要删**，挪到上述
正确位置即可。

### 执行机制

1. **写入时自检**：agent 写任何 markdown 内容前默念 Test A/B。
2. **PR review 关注点**：review 时把"是否越界"作为标准检查项，与
   typo / 死链 / 行数预算并列。
3. **journal 倒查**：发现违规已合入，写 journal 条目，分析是哪个判断
   维度漏了，必要时本 SPEC 加变更记录。
4. **CI 守门（候补，未实现）**：未来可加一个简单 grep 检查（如扫描
   "Twitter handle / 小红书 / 求 star / 加油" 等关键词），但不应过度
   工程化——单作者项目主要靠 SPEC 教育 agent。

## Consequences

- agent 在写任何 markdown 内容前多一道 5 秒检查，写作略慢但显著减少
  返工。
- 长期保持仓库可移交、可 fork、不依赖单一作者人格。
- 现有内容需要回头审一遍——`docs/DEPLOY.md` 已修，其他文件待 baseline
  扫描确认。
- 如果未来需要做推广，能清晰区分"项目活动（在 GitHub 发布 Release notes）"
  和"个人活动（在自媒体公告）"——前者进项目，后者不进。

## Alternatives considered

- **不立明文规则，靠 review 经验把控**：实测无效——上次"自媒体公告"
  就是过了我自己 review 才被用户指出。元规则必须文档化。
- **建立 review checklist 但不写 SPEC**：checklist 没有"为什么"的论述，
  agent 看到不会主动遵守。SPEC 形式有 Context + Decision，agent 在
  context 不足时会引用本 SPEC 自我约束。
- **在 AGENTS.md 详细展开**：违反 AGENTS.md 精简原则。仅在 AGENTS.md
  协作准则保留一行指针。
