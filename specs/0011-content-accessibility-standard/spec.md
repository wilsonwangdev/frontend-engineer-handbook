## Context and Problem Statement

2026-05-21 打磨第 0 / 1 章时，作者本人反思自己对 D1-D5 代号、Tier 抽象
命名、表格 3-5 字缩略单元格都"理解起来有点困难"。这违反 SPEC-0001
明确声明的目标读者——初学者 / 转行者必须读得懂。

问题不是技术性的，是**写作模式缺陷**：抽象概念前没有具体例子、术语
首次出现不解释、缩略表达过度、不同分级体系并存时不说视角差异。
这些都是"懂的人都懂"的内部表述，对目标读者就是门槛。

需要一份**跨章节、跨会话、跨贡献者的写作标尺**，让"通俗易懂"从口号
变成可执行的检查项。同时确保未来 agent 在新会话扫 specs/ 入口表时
能命中。

## Decision Drivers

- 目标读者（初学者 / 转行者）必须读得懂——这是 SPEC-0001 的硬承诺
- 标尺要具体可执行，不能是"写清楚一点"这种模糊原则
- 跨会话 agent 必须能从 specs/ 入口扫到，不依赖 memory
- 与 SPEC-0008 系列一致：不引入自动化校验
- 保留作者判断空间——清单是反射触发器，不是机械约束

## Considered Options

- **A. 立 SPEC（采纳）**：决策记录的标准位置，索引可扫
- **B. 写进 AGENTS.md**：违反 SPEC-0008 R5——AGENTS.md 刚瘦身完
- **C. 写进 docs/CONTENT-STANDARD.md**：docs/ 是参考文件，不是决策位置
- **D. 只靠 user memory**：不跨项目 / 不跨贡献者，已被验证不可靠

## Decision Outcome

选 **A**。本 SPEC 包含三块：

### 1. 5 项写作准则

每写一段内容时主动触发：

1. **抽象前必有具体例子**——抽象概念（如 D1-D5、Tier 分类、心智模型）
   出现前必须先有一个具体场景帮读者建立锚点
2. **术语首次出现必有人话别名**——`agent-ready` → "AI 友好"、
   `reverse changelog` → "反向更新日志"、`D1` → "可验证"
3. **表格单元格不缩略**——3-5 字简写要补成完整短句。
   "diff 即上下文" → "当前 diff 就是全部上下文"。
   读者读完即懂，不需要脑补
4. **分级体系并存时显式说视角差异**——三档（知识）vs 四 Tier
   （任务）必须明说"两者视角不同"，不让读者自己脑补
5. **每段问"新人能否读懂"**——一个零基础读者读到这段会不会卡？
   会就重写
6. **避免用具体数字描述项目当前状态**——"本站有 35 个页面""10 个术语"
   这类数字今天写明天就过期，维护成本高。改用定性表述："所有内容页面"
   "本站采用纯静态生成"

### 2. 每章上线前自检清单（4 问）

> **与 AGENTS.md 的分工**：本清单已 inline 进 [AGENTS.md](../../AGENTS.md)
> §协作准则·1，作为日常写作时的反射触发器。低成本改动（修字 / 调链接 /
> 语气微调）豁免。深度复核（季度盘点、新章节立项、内容争议复核）才打开
> 本 SPEC 全文，对照 5 项准则 + 反例库。

1. 第一段读完，目标读者知不知道这章要讲什么？
2. 出现的术语都有"什么时候被解释"了吗？（首次出现就解释 or 在前置
   章节定义过）
3. 表格里的每一格读完即懂吗？（无需脑补、无需上下来回看）
4. 有没有"懂的人会懂"的内部表述？（inside joke）

任一问答不上 → **重写到能答上为止**。

### 3. 反例库（从过往踩坑积累）

未来 agent 扫这里能避免重蹈覆辙：

| #   | 反例                                                                                                | 修复方向                                                                                                                                                                               |
| --- | --------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | "三档归位"用"浏览器渲染管线 / TypeScript 基础类型"当必学例子                                        | 对新人是黑话；改用"看懂报错信息 / 调试一个 bug"                                                                                                                                        |
| 2   | D1-D5 + Tier 表格直接甩代号 + 3 字缩略单元格                                                        | 配语义别名（D1 可验证）+ 单元格补完整短句                                                                                                                                              |
| 3   | "反向 changelog" / "agent-ready" / "独立自洽" 这类 inside joke                                      | 翻译为"反向更新日志 / AI 友好 / 每节都能单独读"                                                                                                                                        |
| 4   | "三档" 和 "四 Tier" 并存但不说视角差异                                                              | 显式说"三档管知识，四档管任务"                                                                                                                                                         |
| 5   | "时效与勘误" 承诺"半年后显示提示" 但功能未实现                                                      | 文字与实际功能严格对齐，没实现就别写                                                                                                                                                   |
| 6   | 延伸阅读外链没标注访问性，国内读者点开发现需代理才知道                                              | 受限链接旁加「需代理」；第 0 章统一段落说明原则                                                                                                                                        |
| 7   | CSS / 浏览器新特性（如 `text-wrap: pretty`）直接引入未验证中文                                      | 新特性默认为英文设计，中文场景往往反常；引入前必须单独验证；优先用最简的标准属性                                                                                                       |
| 8   | 缩写（a11y / i18n / l10n / WCAG / ARIA / APG / GDS / ADA）首次出现在表格 / 标题里、解释藏在后面段落 | 节首引言段就先约定一行术语表；表格单元格用缩写**之前**段落必须给过完整全称至少一次                                                                                                     |
| 9   | 用 ASCII / Unicode box-drawing 字符（`│ ▼ ┌─┐` 等）画决策树 / 流程图                                | 中英混排（中文字符 2 宽 / 英文 1 宽）+ 不同等宽字体的对齐策略不同 → 大概率错位；改用 markdown 嵌套列表 + 加粗节点（不依赖字体），或用 SVG 组件（参 `src/components/mdx/diagrams.tsx`） |
| 10  | 正文中出现"本站有 35 个页面""10 个术语"等具体数量                                                   | 数字今天写明天就过期——章节增加、术语扩编后需要全文搜索替换。改用定性表述："所有内容页面""采用纯静态生成"。§5.3 已踩此坑并修复                                                          |

新踩坑发现后由作者持续追加。

## Consequences

- **好**：质量基线写入仓库，跨会话 / 跨贡献者一致；新会话扫
  specs/README 索引就能命中
- **好**：5 项准则 + 4 问清单可操作，不是模糊原则
- **不好**：每章上线前多一道自检工序，作者负担增加
- **待还的债**：写第 2 章时按本 SPEC 验证一遍，验证有效性

## Pros and Cons of the Options

| 选项                        | 好处                                     | 代价                                  |
| --------------------------- | ---------------------------------------- | ------------------------------------- |
| **A. SPEC（采纳）**         | 决策记录标准位置；入口表可扫；跨会话有效 | 多一个 SPEC 文件                      |
| B. AGENTS.md                | 第一时间被 agent 看到                    | AGENTS.md 已超阈值；违反 SPEC-0008 R5 |
| C. docs/CONTENT-STANDARD.md | 位置中性                                 | docs/ 是参考资料区，不是决策位置      |
| D. user memory              | 零仓库开销                               | 不跨项目 / 不跨贡献者；已被验证不可靠 |

## Confirmation

如何确认本 SPEC 被遵守：

1. **每章 PR 前作者自查 4 问清单**，结果写进 PR 描述
2. **季度复核**（与 SPEC-0006 / 0010 同周期）：
   - 检查是否有新反例需要补入反例库
   - 评估 4 问清单是否需要调整
3. **不引入自动化**——与 SPEC-0008 一致

  `APG` / `GDS` / `ADA` / `EAA` / `RFC` / `BFC` / `MDN`——这些缩写
  很多都在前几章已经定义过，但每章首次复用时仍要回扫一次"上下文
  里读者真知道这是什么吗"。
- 2026-05-23：追加反例 9（ASCII / Unicode box-drawing 决策图错位）。
  动因：§3.2 §1 用 `│ ▼ ┌─┐` 画"原生 → ARIA → 自造"决策树，中英混
  排在多数等宽字体下错位严重。改成 markdown 嵌套列表 + 加粗节点。
  本规则也覆盖类似的 ASCII 时序图、目录树——能用结构化文本表达的
  场景一律不用 ASCII 美术。SVG 留给真正需要图形（轴向 / 坐标 / 形状）
  的场景。
