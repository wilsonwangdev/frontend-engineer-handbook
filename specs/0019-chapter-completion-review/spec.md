## Context and Problem Statement

第 5 章写作过程中暴露了多个可预防的问题：

- 链接未验证直接写入（Lee Robinson ISR 404、Middleware 已改名 Proxy）
- "并发渲染"标签误标为"所有人"（初学者读不懂）
- §5.3 误写"所有组件默认服务端"导致生产负载风险
- 硬编码数字"35 页面"马上过期
- 编委会立场缺失（为什么推荐 Next.js 没解释）

这些问题有个共同特征：**写完就过去了，没有系统性地回头检查**。
每章写作的教训是宝贵的——但如果只留在对话里，下个会话的 agent
不会知道这些坑。

需要一套**章级完成标准**——写完本章后强制触发定位复核 + 经验记录。

## Decision Drivers

- 整站定位（小而美、不堆砌、最新实践、新手友好）不能因内容膨胀悄悄漂移
- 每章校对中踩的坑必须留存——同一坑不应被不同章节各踩一次
- 流程要轻量——不能变成走形式的 checklist
- agent 跨会话必须能命中——放在 specs/ 索引

## Decision Outcome

每章最后一节上线后，执行以下两步：

### 1. 定位复核（3 问）

| #   | 问题               | 细则                                                                                          |
| --- | ------------------ | --------------------------------------------------------------------------------------------- |
| 1   | **规模是否失控？** | 本章是否在"覆盖必要知识"和"避免变成百科全书"之间取得平衡？有没有可以砍掉的节？                |
| 2   | **立场是否明确？** | 如果本章有主观推荐（如"用 Zustand 而非 Redux"），是否解释清楚了推荐的**理由**和**适用边界**？ |
| 3   | **新手能读懂吗？** | 以"刚学完上一章"的读者视角重新扫一遍第一节——有没有跳步？有没有术语未解释？                    |

任一问答不上 → 先修再标完成。

### 2. 经验积累（journal）

每章完结后写一条 journal，记录：

- 本章写作中踩了哪些坑（内容/流程/工具链）
- 重复出现 ≥ 2 次的坑升级到 `docs/GOTCHAS.md`
- 新发现的写作反例追加到 SPEC-0011 反例库

## 定位基准（SSOT）

复核时对照以下基准文档：

| 基准           | 文件      |
| -------------- | --------- |
| 目标读者与范围 | SPEC-0001 |
| 目录结构       | SPEC-0003 |
| 内容来源准入   | SPEC-0002 |
| 内容可读性标准 | SPEC-0011 |
| 设计判断暗线   | SPEC-0013 |
| 品味培养       | SPEC-0014 |

## Consequences

**好**：

- 章节完成有明确的"发版标准"，不是"写完了就算"
- 跨章经验积累有制度化渠道（journal → GOTCHAS → SPEC-0011 反例库）

**不好**：

- 每章多一道工序
- 3 问是主观判断，不同人可能给出不同答案
