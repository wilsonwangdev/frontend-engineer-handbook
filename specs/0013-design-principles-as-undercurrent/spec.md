## Context and Problem Statement

设计模式、架构分层、模块化这类"软件工程经验"在 AI 时代的价值反而上升——
AI 能高速产出可工作的代码，但**为什么这样切分、为什么用这个模式、什么
时候不要用**这种判断仍然只能由人做。这是 [SPEC-0001](../0001-scope-and-audience/spec.md)
所说"工程师价值的新落点"的具体支柱之一。

但这类话题不适合单独成章：

- 脱离场景讲设计模式，会退化成"23 种 GoF 模式中文翻译"——和 MDN /
  refactoring.guru 这类站点重复，价值低
- 单独章节会让读者觉得"先学完模式再写代码"，违反 [SPEC-0001](../0001-scope-and-audience/spec.md)
  "面向实践、随时翻阅"的定位
- 真正的设计判断永远嵌在具体取舍里——讲 React 状态管理时谈观察者、
  讲 bundler 时谈访问者、讲 monorepo 时谈分层边界，这才是它该出现的
  位置

2026-05-21 会话用户明确表态："融入到整站的内容倾向中，让这些核心
经验贯穿手册以建立完整心智"。需要把这个意图写成可跨会话执行的标尺，
否则下次会话扫 specs/ 入口表时会丢，每章作者都凭直觉决定"要不要点
设计原则"，结果就是有的章浓有的章淡，整站心智断裂。

## Decision Drivers

- **跨会话防遗失**：意图必须被新会话扫入口表捕获（与 [SPEC-0010](../0010-progressive-ssg-distillation/spec.md) 同模式）
- **嵌入而非堆砌**：设计原则跟随具体场景出现，不为讲而讲
- **可读性优先**：受 [SPEC-0011](../0011-content-accessibility-standard/spec.md) 约束——
  抽象前必有具体例子，术语首次出现要解释
- **不增加新写作负担**：复用每章已有的"知识点 → 取舍判断 → 延伸阅读"
  结构，只是在"取舍判断"环节多问一句
- **兜底入口**：分散嵌入需要一处索引收拢，否则读者建不起整体心智

## Considered Options

- **A. 立 SPEC 设暗线 + 索引兜底（采纳）**：每章在合适位置嵌设计原则
  注解，第 10 章末尾收一张"你刚见过的设计判断"索引表
- **B. 独立成章（如新增第 11 章"设计原则与架构思维"）**：违反
  [SPEC-0001](../0001-scope-and-audience/spec.md) 实践定位 + 与业界资料重合
- **C. 完全不立 SPEC，靠每章作者自由发挥**：跨会话失忆，浓淡不一
- **D. 写进 [SPEC-0011](../0011-content-accessibility-standard/spec.md) 反例库**：性质不对——
  那是写作可读性标尺，本 SPEC 是内容深度标尺，混在一起两边都模糊

## Decision Outcome

选 **A**。建立"设计原则暗线"由三层组成：

### 1. 三类内容触点（每章写作时主动触发）

写一段技术内容时，问自己这段是否触及以下三类：

- **设计模式（patterns）**：讲到一个具体技术做法时，它属于哪种通用
  模式？例如 React `useReducer` 是命令模式（command）的影子、Redux
  middleware 链是责任链（chain of responsibility）、bundler 遍历 AST
  是访问者（visitor）。
- **架构分层（layering）**：讲到一个系统切分时，分层依据是什么？
  例如 Next.js Server / Client Components 是按"运行环境"分层、
  通用层 / 产品层（[SPEC-0010](../0010-progressive-ssg-distillation/spec.md)）是按"复用性"分层、
  MVC / MVVM 是按"职责方向"分层。
- **模块化（modularity）**：讲到一个边界划分时，依据是哪种内聚？
  例如功能内聚（feature folder）、领域内聚（domain-driven）、
  技术内聚（按文件类型分目录）——三者各有适用场景，不是越细越好。

**触及任一项 → 用一段话点出**：

- 这是什么模式 / 分层 / 模块化策略（中文 + 英文术语）
- 为什么这个场景适合（替代方案是什么、它们的代价）
- 一句"AI 能写出符合这个模式的代码，但选哪个仍是人的判断"——
  与 [SPEC-0001](../0001-scope-and-audience/spec.md) "AI 时代取舍" 主线呼应

**不要为了点而点**——如果一段技术内容并不真正触及这三类，强行嵌
反而稀释。沉默比凑数好。

### 2. 写作模板：嵌入位置与体量

每章在以下位置自然嵌入（不新增章节结构）：

- **正文行内**：讲到具体技术决策时用 1-3 句话点破，不展开理论。
  例：「React `useReducer` 把"做什么"和"怎么做"分开——这是命令模式
  （command pattern）的影子。AI 能熟练写出 reducer，但**何时该
  从 useState 升级到 useReducer** 是人的判断。」
- **每章末尾速查表（可选）**：如果本章触发了 ≥ 3 个设计判断，加一
  小段"本章涉及的设计判断"汇总，每行一个，关联回正文位置。
- **不允许的形态**：不要写"补充：什么是访问者模式"长方框、不要
  脚注式罗列、不要专门一节讲模式定义——读者要深入查 refactoring.guru
  /《设计模式》原书。

体量约束：单个注解 ≤ 5 行。超过 5 行说明你在讲模式本身而不是判断，
回到 [SPEC-0001](../0001-scope-and-audience/spec.md) 的"导航 + 判断 + 速查"定位。

### 3. 兜底索引：第 10 章末尾汇总表

第 10 章三个 mini 项目（bundler / framework / agent app）天然密集
触发设计判断。在第 10 章末尾加一节「整站设计判断速查」，把贯穿
全书的设计原则注解收拢成一张表：

| 模式 / 分层 / 模块化      | 出现章节          | 场景一句话         | 替代方案                       |
| ------------------------- | ----------------- | ------------------ | ------------------------------ |
| 访问者模式（visitor）     | 10.1 mini bundler | 遍历 AST 做转换    | 责任链：阶段固定时更直观       |
| 单向数据流                | 5.x React 心智    | 状态 → UI 单向推导 | 双向绑定：表单密集场景仍有市场 |
| ...（写作过程中持续追加） |

这张表是兜底入口——不强求一次写齐，每章新增注解时同步追加一行。
读者从这张表反向跳回正文场景，比从模式定义出发学习更贴近真实判断。

### 4. 触点案例库（持续追加）

仅有"触发器原则"作者会不知从哪起步。下面是起手已识别的子话题，
作者写到对应章节时优先触发本 SPEC §1 的注解机制。**这是种子清单，
不是穷举**——发现新子话题持续追加。

| 子话题                                      | 归属类别      | 建议出现章节                                         | 一句话价值                                                                   |
| ------------------------------------------- | ------------- | ---------------------------------------------------- | ---------------------------------------------------------------------------- |
| 循环依赖（cyclic dependencies）             | 分层 + 模块化 | 第 6 章工程化、第 10.1 mini bundler                  | 分层 / 模块化不合理的高频后果；AI 从单文件视角看不出，需人从依赖图整体识别   |
| 深模块 vs 浅模块（deep vs shallow modules） | 模块化        | 第 5 章组件设计、第 10.2 mini framework              | 评估模块质量的核心维度——接口表面积小、内在功能强即"深"；浅模块易堆叠成抽象税 |
| 接口设计（API design / 信息隐藏）           | 模块化        | 第 5 章组件 props、第 10 章 mini bundler / framework | API 边界一旦发布就难改；是设计判断中"最贵"的一类决策                         |

**深 / 浅模块**与**信息隐藏**的一手来源是 John Ousterhout
[《软件设计哲学》（A Philosophy of Software Design）](https://web.stanford.edu/~ouster/cgi-bin/aposd.php)
第 4 / 5 章。**循环依赖**的检测与拆解可参考社区工具如
[madge](https://github.com/pahen/madge)（依赖图可视化）+
[依赖倒置原则（DIP）](https://en.wikipedia.org/wiki/Dependency_inversion_principle)。
具体素材在写章节时按 [SPEC-0002](../0002-content-source-admission/spec.md)
准入策略 + [SPEC-0011](../0011-content-accessibility-standard/spec.md)
反例 6 访问性核实。

### 5. 与第 1 章「AI 可委托清单」的呼应

第 1 章已有"AI 可委托清单"主线（[SPEC-0001](../0001-scope-and-audience/spec.md) §AI 时代取舍）。
本 SPEC 不强行在第 1 章再加一节"AI 委托不动的设计判断"——避免该章
变重。设计判断的"AI 替代不了"立场通过**每章嵌入的注解**自然累积，
读者读完整本手册自然形成印象，比一次性宣告更有说服力。

如果未来某次复核发现读者反馈"还是没建立心智"，再考虑第 1 章末加
一段"反向清单"压舱。当前不做。

## Consequences

- **好**：意图写入 specs/，跨会话不丢
- **好**：每章作者有明确的"该不该点设计原则"触发器，不再凭直觉
- **好**：读者读完手册自然累积起设计判断的整体感，符合用户表达的
  "建立完整心智"
- **好**：与 [SPEC-0001](../0001-scope-and-audience/spec.md) §AI 时代取舍 + [SPEC-0011](../0011-content-accessibility-standard/spec.md) §可读性
  共同构成内容深度的三层标尺：定位 / 写法 / 深度
- **不好**：每章作者多一道"是否触及三类内容"的反射检查
- **不好**：第 10 章末尾索引表在前期会很短，看起来"空"——这是必要
  的过程，不要为了填满而填
- **待还的债**：写第 2 章时按本 SPEC 验证一遍，看注解是否真的嵌得
  进去 + 不破坏可读性

## Pros and Cons of the Options

| 选项                           | 好处                                     | 代价                                                      |
| ------------------------------ | ---------------------------------------- | --------------------------------------------------------- |
| **A. 暗线 + 索引兜底（采纳）** | 跨会话不丢；嵌入贴近实战；与现有结构兼容 | 作者多一道反射检查；索引前期看起来空                      |
| B. 独立成章                    | 系统性强、读者易找                       | 与业界资料重合；违反 SPEC-0001 实践定位；增加一章打磨成本 |
| C. 不立 SPEC，自由发挥         | 零结构开销                               | 跨会话失忆；浓淡不一；建不起整站心智                      |
| D. 并入 SPEC-0011              | 复用现有 SPEC                            | 性质混淆——可读性 vs 深度是两件事；两边都被稀释            |

## Confirmation

如何确认本 SPEC 被遵守：

1. **每章 PR 前作者自查**：commit body 或 PR 描述简短回答——
   "本章触发了哪些设计判断注解？放在哪几节？"。零注解也算合格答案
   （不强求每章都有），但要显式说明"本章未触及三类内容"
2. **第 10 章末尾索引表持续追加**：每次新章节嵌入注解时，同步追加
   一行到速查表
3. **季度复核**（与 [SPEC-0006](../0006-agent-skills-system/spec.md) /
   [SPEC-0010](../0010-progressive-ssg-distillation/spec.md) /
   [SPEC-0011](../0011-content-accessibility-standard/spec.md) 同周期，
   2026-08 首次）：
   - 检查嵌入的注解是否真的触发了"判断"而非"定义罗列"
   - 评估第 10 章索引表是否需要重组（按模式 / 按章节 / 按场景）
   - 评估 §4 触点案例库是否需要新增子话题（如错误边界 / 重试策略 /
     性能预算等也是设计判断密集区）
   - 评估是否需要在第 1 章补"AI 委托不动的设计判断"反向清单
4. **不引入自动化校验**——与 [SPEC-0008](../0008-pre-action-reflexive-checklist/spec.md) /
   [SPEC-0011](../0011-content-accessibility-standard/spec.md) 一致

### 边界澄清

本 SPEC **不是**：

- 写一份"23 种设计模式中文清单"——那是 refactoring.guru 的事
- 给每个技术决策强加模式标签——沉默比凑数好
- 把手册变成软件工程理论书——主线仍是前端实践

本 SPEC **是**：

- 一份"该在哪点破设计判断"的反射触发器
- 一条把分散注解收拢为整体心智的兜底通道
- 与 [SPEC-0001](../0001-scope-and-audience/spec.md) AI 时代取舍主线呼应的内容深度标尺

### 触发本 SPEC 的会话

2026-05-21 会话讨论"设计模式、架构分层、模块化等高级话题"在 AI
时代愈发重要、但不适合单独学习的内容规划问题。用户表态"融入到
整站的内容倾向中、贯穿手册建立完整心智"。本 SPEC 把这个意图
锁定为跨会话标尺。
