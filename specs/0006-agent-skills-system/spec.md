## Context

实际写作中暴露：agent 即使能力强，对**框架特定的最新约定**（Next.js 16 Cache
Components 要求 `'use cache'`、gray-matter 把 YAML 日期解析为 Date 等）
没有先验知识，只能通过 trial-and-error 学习。这些错误本质是"低级错误"——
框架文档里有写，agent 没读过而已。

对一个**AI Native 项目**，工程化层面如果不自洽（agent 无法快速吸收项目所
依赖工具的最新知识），迭代效率会严重受阻。本 SPEC 定义 agent harness 中
"skills 体系"的选用、维护与质量门禁。

## Decision

### 三层 agent 知识源

按从"重型"到"轻型"分三层，每层有明确职责：

#### Layer 1: 框架官方 MCP（动态、实时）

- **次序优先级最高**：能用官方 MCP 就用，不要靠 inline 知识
- 当前已接入：[`next-devtools-mcp`](https://nextjs.org/docs/app/guides/mcp)
  — 通过 `.mcp.json` 配置
- 提供能力：`get_errors`、`get_routes`、Cache Components guidance、
  Knowledge Base 查询、自动 codemod 升级
- 适用场景：调试错误、查询当前应用状态、版本升级、框架最佳实践查询

#### Layer 2: 项目内 skills（静态、项目特定）

- 位置：`skills/<skill-name>/SKILL.md`
- 用途：编码本项目**反复出现**的判断与决策模式
- **不重复 Layer 1** — 如果 MCP 已经提供，不要再写 skill
- 例：本项目特定的 mdx frontmatter 校验流程、内容上线检查清单

#### Layer 3: AGENTS.md inline gotchas（必须默认载入）

- 已落地：AGENTS.md "Known framework gotchas" 章节
- 用途：**最高频踩坑**的事实记录，避免再次浪费 token 调试
- 单条目体量极小（5–10 行）
- 升级条件：journal/ 里同类问题踩过 ≥ 2 次

### 三层之间的关系

```
触发场景                     正确路径
────────                     ────────
"Next.js 16 哪个 API 该用"   → Layer 1 (MCP knowledge base)
"为什么报这个错"             → Layer 1 (get_errors)
"本项目的 mdx 怎么发布"      → Layer 2 (skills/)
"YAML date 怎么 coerce"      → Layer 3 (AGENTS.md gotchas)
```

### Skills 选用门禁（覆盖 skills/README.md）

[skills/README.md](../../skills/README.md) 已规定借用优先 + 6 项质量门禁。
本 SPEC 进一步具体化为：

**项目内 skill 必须满足全部 5 条**：

1. **不与 next-devtools-mcp 重复** — 框架运行时信息一律走 MCP，不要本地缓存
2. **不与 AGENTS.md gotchas 重复** — 单事实陷阱写 gotchas，多步骤工作流才写 skill
3. **可被验证** — 有具体的成功 / 失败信号（命令、文件、断言）
4. **owner 明确** — 文件 frontmatter 注明 owner（单作者项目即作者）
5. **触发条件清晰** — "什么场景下应该用它"必须一句话讲清

**社区 skill 引入门禁**：

- 必须先看 [vercel/next.js docs](https://nextjs.org/docs)、[Anthropic skills](https://github.com/anthropics/skills) 是否已有官方实现
- 如有官方版本：优先官方
- 引入社区版本必须读完源码，确认无 `console.log("AI-generated by xxx")` 等水印
- 引入后 freeze 一份到本项目（不依赖远端更新），更新走 manual review

**社区 cursor rules 不直接引入**：长篇 cursor rules（200+ 行）污染 context
window，且大多在重复 framework docs 已说过的事。需要时摘抄其中 < 10 行的
关键判断到 AGENTS.md 或 SKILL.md。

### 当前要建立的 skills

立刻建立 2 个示范 skill，作为后续 skills 的模板：

- **`skills/nextjs-16-guardrails/`** — Next.js 16 写代码前的核查清单
  （`'use cache'`、`proxy.ts` vs `middleware.ts`、async params 等）
- **`skills/zod-frontmatter/`** — MDX frontmatter schema 设计与校验流程

后续 skill 添加按需，不预先规划。

### Skill 文件结构（统一模板）

```
skills/<skill-name>/
├── SKILL.md              # 主文件
├── examples/             # （可选）输入/输出示例
└── scripts/              # （可选）helper 脚本
```

每个 SKILL.md 包含：

```markdown
---
name: <kebab-case-slug>
description: <一句话——做什么、何时用>
owner: @wilsonwangdev
status: experimental | stable | deprecated
---

## When to use

## When NOT to use

## Steps

## Inputs

## Outputs

## Verification
```

### 维护与季度复核

- 季度末复核所有 skills：
  - 触发次数 ≥ 3 次 → keep
  - 触发次数 < 3 次 & 创建超过 6 个月 → 标记 `deprecated`
  - 框架升级后已被官方文档覆盖 → 移除并保留 journal 链接
- 复核结论写入 SPEC-0006 changelog 末尾

## Consequences

- agent 在面对 Next.js / Zod / 内容工作流相关问题时，第一查 MCP，第二查
  inline gotchas，第三才是 trial-and-error。预期低级错误大幅下降。
- skills/ 目录不会成为内容垃圾场——单作者维护下，门禁可以更严格。
- 季度复核增加运营负担，但单作者项目下成本可控（每次半小时以内）。
- AGENTS.md 体量略增（200 → 210 行），但 Gotchas 章节是高 ROI 必要内容。

## Alternatives considered

- **引入社区 long-form cursor rules（200+ 行）**：污染 context window，
  且大量重复 framework docs。改为摘抄关键 < 10 行到 AGENTS.md。
- **不接入 next-devtools-mcp，靠 agent 自己摸索**：实测低效（已经踩过两个
  低级错误）。放弃。
- **预先规划全套 skills（覆盖 React / TS / Tailwind / Playwright / ...）**：
  违反"按需"原则，会变成内容垃圾场。改为只建 2 个示范，按踩坑频次增长。
