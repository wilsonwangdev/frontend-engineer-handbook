## Context

写作和工程化过程中持续浮现两类内容，主干 10 章吸纳不下，但又有真实价值：

1. **与 agent 协作时的"框架特定坑"** —— 例如 Next.js 16 Cache Components 要求 `'use cache'`、gray-matter 把 YAML 日期解析为 `Date` 对象等。
2. **工程化生态的"非业务问题"** —— 例如 Playwright + 系统级 TUN 模式代理的网络拦截、stray dev server 占用端口、`http_proxy` 环境变量对 webServer health-check 的污染。这些本质是操作系统 / 网络栈层面的问题，但只有前端工程师在用这些工具，且**对中文工程师额外不友好**（代理、镜像源、Clash/Surge/TUN 模式）。

[SPEC-0003](../0003-table-of-contents/spec.md) 的主干结构（10 章）刻意做减法，把这类内容放进去会破坏"小而美"定位。但完全不收，又错过了手册"对读者负责"的机会——大多数读者在跑通工程化流程时，都会在这些没规律的问题上浪费大量时间。

同时，写作过程中也暴露了另一个**品味决策**：内容类页面的 E2E 测试，agent 默认会用"具体文字断言"（`expect(getByText("Tier 1：完全委托"))`）——这种风格让任何内容更新都触发测试失败，本末倒置。需要在测试策略上立约定。

## Decision

### 一、附录 D：实战避坑录

在主干 10 章 + 附录 A/B/C 之后，新增 **附录 D 实战避坑录**。

#### 定位

| 维度     | 主干 10 章               | 附录 D                               |
| -------- | ------------------------ | ------------------------------------ |
| 内容性质 | AI 时代必学路径          | AI 时代避坑指南                      |
| 三档归位 | 必学 / 仍需理解 / 可委托 | **全部默认"仍需理解"**，不进必学     |
| 来源     | 一手权威 + 经典书目      | 本站实战积累 + 一手权威辅助          |
| 阅读方式 | 系统读 / 速查 / 实践驱动 | **纯查询式**，遇到再读               |
| 体量上限 | 单章 200–400 行          | **单条目 ≤ 200 字**（含 workaround） |
| 时效预期 | 半衰期 ≥ 2 年            | 半衰期 ≥ 1 年（更短即不收）          |

#### 收录范围

附录 D **只收**满足以下 4 项条件的避坑案例：

1. 影响 ≥ 10% 的中文前端工程师（不是 niche 问题）
2. 主流文档 / 中文搜索引擎里找不到清晰中文答案
3. 能给出 5 分钟内可执行的 workaround
4. 不会很快被框架 / 工具自身修复（短半衰期则不收）

通过 4 项的案例进入正式条目，不通过的留在 `journal/` 即可。

#### 不收的范围

显式排除：

- 普通 bug（去 issue tracker 报告即可，不进手册）
- 个人配置问题（特定 IDE / shell / 终端的偏门设置）
- 仅在特定操作系统某个版本出现的临时性问题
- 主流文档里已有清晰中文版的（重复劳动）

#### 条目格式

每个条目控制在 ≤ 200 字（不含代码），分三段：

```markdown
## D.N 标题（一句话说明症状）

**症状**：你看到什么、什么时候看到、报错信息原文（可被搜索）。

**Workaround**：5 分钟内可执行的步骤，最好是 1–3 行代码或配置。

**原理**（一句话）：为什么会这样、哪个层级的问题、长期会不会变。

**完整事故记录**：[journal/YYYY-MM-DD-...](../../../journal/...)
```

#### 与 journal/ 的关系

附录 D 是 journal/ 的精选 + 提炼，不是替代关系：

- **journal/** 记录全部踩坑细节（首发地）
- **附录 D** 是经过 4 项门禁的精选条目（对外的查阅入口）
- 二者通过链接互通：journal 完整故事，附录 D 速查 workaround

#### 分类（初步）

附录 D 内部按以下分类组织：

- **D.1 网络与代理**（中文环境特有问题：TUN 模式、镜像源、SSL 证书）
- **D.2 框架特定坑**（Next.js / React / TypeScript 等版本敏感问题）
- **D.3 工具链协同**（Playwright / Vitest / 构建工具 / lint 工具）
- **D.4 部署与生产环境**（Vercel / 容器化 / CI/CD）
- **D.5 macOS / Windows / Linux 差异**（仅当影响开发体验时）

### 二、第 7 章测试策略约定

[SPEC-0003](../0003-table-of-contents/spec.md) 已规定第 7 章覆盖"测试金字塔：Vitest（单元）/ Playwright（E2E）"。本 SPEC 进一步约束**测试策略的写作内容**：

第 7 章必须包含一节 **"内容站 vs 应用站的测试策略对比"**，明确：

#### 内容类页面（如本手册）

- ✅ **快照测试**主流（结构 + 渲染输出）
- ✅ **结构性断言**（"存在 h1 标题"、"渲染了 5 个章节卡片"）
- ❌ **具体文字断言**（"包含『Tier 1：完全委托』"）—— 反例

理由：内容是产品本身且高频更新，具体文字断言把测试变成"内容审查工具"，
让小幅措辞调整也触发测试失败，违背 D3（错误代价低）。

#### 应用类页面（交互、状态、API）

- ✅ **行为断言**（"点击后 navigate 到 /xxx"、"提交表单触发 POST"）
- ✅ **可访问性断言**（axe-core / WCAG 标签）
- ⚠ **快照**仅用于稳定的视觉回归

#### 本项目的具体应用

- `e2e/home.spec.ts`、`e2e/handbook.spec.ts` 现在用了具体文字断言——
  **这是反例**，第 7 章动笔时改造为快照 + 结构断言，并作为案例对比展示。
- 改造前后的 diff 收入附录 D 或第 7 章的"案例研究"小节。

### 三、产出工作流

#### journal/ → 附录 D 的升级路径

1. 遇到坑 → 立即写 `journal/YYYY-MM-DD-slug.md`（已经在执行）
2. 季度末复核 journal/ → 对照附录 D 的 4 项门禁筛选
3. 通过的条目 → 写进 `content/appendix-d/sections/<category>.mdx`
4. journal/ 里加 `promoted-to: appendix-d#D.X` 字段
5. 同步更新本 SPEC 的 Changelog

#### 写作时的 hook

写主干章节遇到"想顺便讲一下这个坑"时——**不要塞进主干**，直接：

1. 在 journal/ 写一条
2. 在主干章节加一行：`> 该坑详情见 [附录 D.X](/appendix-d#D.X)`

这样主干保持精简，番外承接细节。

## Consequences

- 附录 D 给"非主干价值内容"找到了归宿，避免主干膨胀。
- 4 项门禁 + 200 字上限让番外不会变成内容垃圾场。
- journal → 附录 D 的工作流让真实踩坑自然沉淀为公共资产。
- 第 7 章测试策略一节直接以本项目当反例，强化"吃自己狗粮"的承诺。
- 现有 e2e 测试需要在第 7 章写作时回头改造（短期负债，要在 TODO 里跟踪）。
- 番外章节本身不进入"系统学习路径"（路径 A）。仅在路径 B（速查）和实践遇坑时被访问。

## Alternatives considered

- **塞进第 9 章 AI 原生工作流**：融合度高但有膨胀风险，且番外内容大多不是 AI 工作流问题（HTTP 代理就不是）。放弃。
- **不做附录 D，全留在 journal/**：journal/ 对内是日志，对外不友好（没有结构化分类、没有 4 项门禁筛选）。放弃。
- **附录 D 容纳所有踩坑记录**：违反"小而美"。门禁是必须的。
- **测试策略不在主干约定，让作者自由发挥**：实测发现 agent 默认风格是反例，必须立约束。放弃。
