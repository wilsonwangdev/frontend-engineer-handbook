## Context and Problem Statement

2026-06-04 起站点从"内容主导"向"内容 + 公共能力"演化——
新增了 `/glossary`（术语表）和未来的 `/playground`（动手玩集中页）。
但顶部 header 当前只有 logo、阅读宽度切换、GitHub 链接——**这两个新公共页面没有主动入口**。

读者只有从内容里的 `<Term>` 浮窗"查看详情"或末尾跳转链接进入这些页面——
完全被动。新读者不会知道站点除了章节还有什么。

需要决策：

- header 怎么加新入口
- 移动端抽屉怎么同步
- 新页面之间的相互可达性（如 `/glossary` 与 `/playground`）
- 何时加（新增公共页时）/ 何时不加（一次性页面）

约束：

- 不能让 header 拥挤——已有 logo / GitHub / 阅读宽度切换三个固定项
- 移动端 viewport 极窄——header 字符预算 < 30 个汉字宽
- 现有 `MobileNav` 抽屉是章节导航——加公共入口不能淹没章节
- 公共页面会逐步增加（搜索 / 路径选择器 / 进度等都在 ROADMAP）

## Decision Drivers

- 新读者第一眼能发现"站点不只是章节"
- 桌面 / 移动端体验统一
- 扩展友好——加新公共页时改一处即可
- 不破坏现有 sidebar 的章节导航地位

## Considered Options

### 选项 A：header 直接加顶级链接（推荐）

桌面端 header 中央或右侧加一组导航：

```text
┌─────────────────────────────────────────────────┐
│  [前端工程师手册]  词条  Playground   ☀️  GH    │
└─────────────────────────────────────────────────┘
```

移动端抽屉顶部加"公共页面"分组（在章节导航之上）。

**优点**：直接、清晰；新读者一眼可见。
**缺点**：header 元素增加；需要约束最大入口数（< 4 个）。

### 选项 B：单独的"更多"下拉菜单

```text
┌──────────────────────────────────────┐
│  [前端]  目录  更多 ▾  ☀️  GH        │
└──────────────────────────────────────┘
              ↓
        ┌──────────┐
        │ 词条     │
        │ Playground │
        │ 搜索     │
        └──────────┘
```

**优点**：扩展性最强（容纳无限项）；header 不挤。
**缺点**：多一层点击；新读者不容易发现。

### 选项 C：footer 集中

把所有公共页面入口放 footer。

**优点**：header 完全保持现状。
**缺点**：滚动到底部才能看到——发现率最低。

## Decision Outcome

选 **A**。理由：

- 站点核心交互是阅读章节——新公共页若隐藏在二级菜单里发现率太低
- v1 阶段公共页面数量可控（≤ 4 个：词条、Playground、搜索、路径选择器）
- 未来超过 4 个再演化为 B 方案

## 实施细节

### 桌面端 header

| 位置 | 元素                                    | 何时显示          |
| ---- | --------------------------------------- | ----------------- |
| 左   | Logo + 移动端汉堡                       | 始终              |
| 中   | 词条 / Playground / ...（公共页面入口） | 桌面端（≥ 768px） |
| 右   | 主题切换 / 阅读宽度切换 / GitHub        | 始终              |

最大公共入口数：**4 个**。超过时升级到选项 B（下拉）。

### 移动端 `MobileNav` 抽屉

抽屉打开时顶部加"公共页面"分组：

```text
┌─────────────────┐
│ [×]             │
│                 │
│ 公共页面        │
│   词条          │
│   Playground    │
│ ──────────────  │
│ 章节            │
│   00 如何使用   │
│   01 ...        │
└─────────────────┘
```

### 公共页面之间的关联

`/glossary` 和 `/playground` 互相在 footer 列出——读者从一个页面能直达另一个。

### 当前活动状态视觉

进入 `/glossary` 时 header 上"词条"高亮（accent 颜色）。`usePathname()` 判断。

## Consequences

**好**：

- 新读者发现率提升
- 公共页面有"被认真对待"的视觉暗示（不只是埋在某个段落里的链接）
- 未来加搜索 / 路径选择器时直接复用这套机制

**不好**：

- header 入口数有上限——超过 4 个要重构为选项 B
- 桌面端 header 视觉密度上升
- 需要维护"哪些页面算公共页面"的清单

**待还的债**：

- 当公共页面 ≥ 5 个时升级到选项 B（下拉菜单）
- 移动端窄屏（< 360px）若 header 中央放不下入口，需要降级为汉堡内露出

## 实施触发条件

本 SPEC 是 `proposed` 状态。下一步实施触发条件：

1. **立即实施**：`/glossary` 已上线但无导航入口——需要修复
2. **同时实施**：本次 commit 同步加 Playground 入口（即使 `/playground` 还没建——
   先做"占位 + 链接到 `/playground` 路由"，路由在 SPEC-0016 上线时同步真实化）
3. **延后实施**：搜索 / 路径选择器入口——这些功能落地时再加

## 验收标准

- 桌面端 header 出现"词条"链接，点击进入 `/glossary`
- 移动端抽屉打开后能看到"公共页面"分组
- 当前活动页面对应的入口高亮
- header 在 320px / 768px / 1280px 三档宽度下视觉合理
- a11y：键盘可访问，`aria-current="page"` 正确

## Pros and Cons of the Options

### A：header 直接加顶级链接

- 好：发现率高 / 实施简单 / 与现有 header 模式一致
- 不好：扩展性受限于横向空间（≤ 4 个入口）

### B：下拉菜单

- 好：扩展性无上限
- 不好：多一层点击 / 新读者发现率降低

### C：footer 集中

- 好：header 不变
- 不好：发现率最低 / 与"阅读为主"的用户路径不匹配
